# Архитектурные решения (ADR)

## 1. Union-Find для кластеризации имён проектов

**Проблема:** Один проект может называться по-разному в 4 источниках данных (P&L, расходы, доходы, выплаты). Простой словарь не решает задачу транзитивного замыкания: если "Project A" совпадает с "Project A (v2)", а "Project A (v2)" — с "ProjectA", все три должны быть в одном кластере, даже если прямое сравнение A↔ProjectA не срабатывает.

**Решение:** `ProjectRegistry` использует алгоритм Union-Find с 3-уровневым матчингом (`ProjectMatchingService.AreMatched()`). Для каждой пары нормализованных имён проверяется совпадение → один кластер.

**Компромисс:** O(n² · α(n)) сложность при построении. Приемлемо для <200 проектов — построение занимает миллисекунды.

**Файл:** `BusinessAnalytics.Core/Services/ProjectRegistry.cs`

## 2. Два оркестратора: Console (10 стадий) vs Web (5 стадий)

**Проблема:** Нужны два режима работы — ручная диагностика и автоматический запуск с мониторингом.

**Решение:**
- **RunAnalytics (Console)** — 10 отдельных стадий. Каждая верификация — своя стадия. Гранулярный вывод: видно, какая именно проверка упала и с какими данными.
- **ParsingPipelineApp (Web)** — 5 укрупнённых стадий. Все верификации в одной `VerificationStageAsync`. Прогресс отображается в React UI, результаты сохраняются в SQLite.

**Компромисс:** Частичное дублирование логики между sync/async реализациями стадий. Оправдано разной моделью вывода (консоль vs БД + REST API).

**Файлы:** `RunAnalytics/Program.cs`, `ParsingPipelineApp.Api/Services/PipelineOrchestrator.cs`

## 3. Strategy — IDataSource для абстракции источников

**Проблема:** Парсеры не должны зависеть от конкретного способа чтения данных (Google Sheets API vs JSON файл vs будущие источники).

**Решение:** Интерфейс `IDataSource` с единственным методом `ReadDataAsync()` → `TableData`. Две реализации:
- `GoogleSheetsDataSource` — чтение через Google Sheets API v4
- `JsonFileDataSource` — чтение из JSON файлов (для тестов и предзагруженных данных)

Все 5 парсеров принимают `IDataSource` через конструктор. Данные приходят в унифицированном формате `TableData` с адресацией ячеек ("A1", "B5").

**Компромисс:** Один дополнительный уровень абстракции. Окупается тестируемостью — тесты используют `JsonFileDataSource` без обращений к Google API.

**Файлы:** `BusinessAnalytics.Core/Abstractions/IDataSource.cs`, `BusinessAnalytics.Core/DataSources/`

## 4. Template Method — BaseTableParser для 5 парсеров

**Проблема:** 5 парсеров разных листов решают одинаковые низкоуровневые задачи: конвертация адресов ячеек в индексы, парсинг decimal/дат с учётом локали, поиск заголовков в переменных позициях.

**Решение:** `BaseTableParser` содержит общие утилиты:
- `GetCellValue()`, `ExtractColumnLetter()`, `GetColumnIndex()` — адресация
- `ParseDecimal()`, `GetCellDateValue()` — типизация
- `FindColumns()` — поиск заголовков

Каждый парсер наследует базовый класс и реализует только доменную логику (блочные маркеры в `ExpensesParser`, многострочные проекты в `ProjectPLParser`).

**Компромисс:** Наследование создаёт связанность. Смягчается тем, что базовый класс содержит только утилиты, не навязывая структуру парсинга.

**Файл:** `BusinessAnalytics.Core/BaseTableParser.cs`

## 5. SemaphoreSlim(1,1) — запрет параллельных запусков pipeline

**Проблема:** Web API допускает одновременные HTTP-запросы. Параллельный запуск pipeline вызовет: конфликты записи в БД, перезапись файлов результатов, race condition при построении ProjectRegistry.

**Решение:** `SemaphoreSlim(1, 1)` с неблокирующей проверкой `WaitAsync(0)`:
```csharp
if (!await _executionLock.WaitAsync(0))
    throw new InvalidOperationException("Pipeline is already running...");
```

Второй запрос получает мгновенный отказ (не ждёт в очереди). Блок `finally` гарантирует освобождение lock при любом исходе.

**Компромисс:** Только один pipeline за раз. Приемлемо для аналитического инструмента — запуск раз в день/неделю.

**Файл:** `ParsingPipelineApp.Api/Services/PipelineOrchestrator.cs`
